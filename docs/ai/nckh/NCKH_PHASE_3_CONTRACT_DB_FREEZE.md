## Freeze Decision: ModelRelation

Approval status: **Proposed for Phase 3 implementation approval**.

Entity fields:

- `Id` GUID primary key
- `ModelId` GUID FK to `ResearchModels.Id`
- `FromVariableId` GUID FK to `ResearchVariables.Id`
- `ToVariableId` GUID FK to `ResearchVariables.Id`
- `Direction` string, allowed values: `Positive`, `Negative`
- `HypothesisCode` string, generated by the service
- `HypothesisText` string, generated by the service
- `SortOrder` int
- `CreatedAt` DateTimeOffset
- `UpdatedAt` DateTimeOffset

Database rules:

- `ModelId -> ResearchModels`: cascade delete
- `FromVariableId -> ResearchVariables`: restrict delete
- `ToVariableId -> ResearchVariables`: restrict delete
- unique index on `(ModelId, FromVariableId, ToVariableId)`
- unique index on `(ModelId, HypothesisCode)`
- check constraint rejecting `FromVariableId == ToVariableId`
- explicit EF Core configuration for both FKs to `ResearchVariables`

Validation rules:

- the model must belong to the current user
- both variables must belong to the same model
- self-relation is rejected
- duplicate `(ModelId, FromVariableId, ToVariableId)` is rejected
- inverse relation is allowed because it represents a different direction of influence
- relations may be created, updated, or deleted only while the model is `Draft`
- `Active` models are read-only for Phase 3 relation edits

Rationale:

- This keeps Phase 3 aligned with the existing `Draft -> Active` lifecycle without introducing `Archived`.
- Restrict delete on variables prevents silent loss of hypothesis relations when a variable is deleted.
- Unique source-target pairs avoid contradictory positive/negative duplicates for the same directed relation.

## Freeze Decision: NodePosition

Approval status: **Proposed for Phase 3 implementation approval**.

Entity fields:

- `Id` GUID primary key
- `ModelId` GUID FK to `ResearchModels.Id`
- `NodeType` string, allowed values: `Variable`, `Relation`
- `VariableId` GUID nullable FK to `ResearchVariables.Id`
- `RelationId` GUID nullable FK to `ModelRelations.Id`
- `PositionX` decimal with precision `(18, 2)`
- `PositionY` decimal with precision `(18, 2)`
- `CreatedAt` DateTimeOffset
- `UpdatedAt` DateTimeOffset

Database rules:

- `ModelId -> ResearchModels`: cascade delete
- `VariableId -> ResearchVariables`: cascade delete
- `RelationId -> ModelRelations`: cascade delete
- check constraint: exactly one of `VariableId` or `RelationId` must be set
- unique index on `(ModelId, NodeType, VariableId)` filtered where `VariableId IS NOT NULL`
- unique index on `(ModelId, NodeType, RelationId)` filtered where `RelationId IS NOT NULL`

Validation rules:

- the model must belong to the current user
- variable positions must reference variables in the same model
- relation positions must reference relations in the same model
- positions may be saved only while the model is `Draft`
- `Active` models are read-only for Phase 3 position edits
- coordinate values are accepted as decimal values with two persisted decimal places
- no viewport bounds are enforced in Phase 3 backend

Rationale:

- Store both variable and relation nodes because the proposed API already includes both node types.
- Avoid React Flow-specific payload ownership in the backend while preserving enough coordinate data for a future UI.
- No backend viewport bounds avoids hardcoding frontend canvas dimensions.

## Freeze Decision: Hypothesis Generation

Approval status: **Proposed for Phase 3 implementation approval**.

Rules:

- hypothesis generation is deterministic only
- no AI provider calls
- `HypothesisCode` is generated as `H{n}` within the model, ordered by relation `SortOrder` then creation time
- `Positive` text template: `{fromVariableName} has a positive influence on {toVariableName}`
- `Negative` text template: `{fromVariableName} has a negative influence on {toVariableName}`
- Vietnamese display wording can be handled by frontend/localization later; the persisted backend text stays deterministic

Rationale:

- Keeps Phase 3 testable and repeatable.
- Avoids approving AI-generated hypothesis behavior.
- Keeps statistical interpretation out of the app.

## Error Behavior

Use the existing Phase 2 service-result pattern unless a future API-wide error format refactor is separately approved.

Expected status mapping:

- `400` validation failure, invalid direction, self-relation, invalid node reference, invalid node type
- `401` unauthenticated request
- `404` model, variable, relation, or position target not found in the current user's scope
- `409` duplicate directed relation or duplicate node position conflict

## DB Risk Review

Findings:

- High: dual FK from `ModelRelations` to `ResearchVariables` must be explicitly configured with `DeleteBehavior.Restrict`; otherwise EF Core migration generation or SQL Server cascade path behavior can fail.
- High: variable delete must be blocked while relations exist, otherwise hypothesis relations can be silently lost.
- Medium: filtered unique indexes for nullable `VariableId` and `RelationId` need SQL Server-compatible filters.
- Medium: hypothesis code generation can race if two relations are created concurrently for the same model; implementation should either generate inside a transaction or rely on the unique `(ModelId, HypothesisCode)` index and return conflict on collision.
- Medium: editing `Active` models could change a research model after it has been activated; Phase 3 should keep edits Draft-only.
- Low: coordinate values are UI-oriented and should not carry business meaning.

Required implementation safeguards:

- migration must be reversible
- migration must not alter Phase 1 or Phase 2 tables except adding reviewed FKs from Phase 3 tables
- service tests must cover ownership, cross-model variables, self-relation, duplicate relation, and Draft-only edit rules
- runtime HTTP smoke must cover relation CRUD and position save/load before closeout


## Approved Implementation Adjustment

During Phase 3 implementation validation, SQL Server rejected the original all-cascade `NodePositions` FK shape because it created multiple cascade paths. The user approved the implementation adjustment to use restrict/no-action for `NodePosition.ModelId -> ResearchModels` and `NodePosition.VariableId -> ResearchVariables`, while preserving cleanup behavior in services and keeping `NodePosition.RelationId -> ModelRelations` cascade. See `NCKH_PHASE_3_CLOSEOUT.md`.
