# NCKH_PHASE_4_KICKOFF_PLAN

## Purpose

Define the approval-gated kickoff plan for NCKH Phase 4 so Google Form Generation & Update work can proceed without widening into data collection, normalization, export, frontend expansion, or NCKH credit/pricing.

This document is a planning and handoff artifact. It does not mark Phase 4 as implemented or completed.

## Phase Goal

Open the next post-Phase-3 implementation slice for the NCKH module:

- generate a Google Form from an owned research model and its mapped variables/questions
- update a Google Form only through an approved ownership-safe flow
- re-import the resulting Google Form structure into existing `ResearchForm` / `ResearchFormQuestion` records after successful create/update

Phase 4 is a Google Forms write phase. It does not approve Google Sheets response collection, normalization, export, credit, pricing, NCKH admin UI, statistical analysis, scheduled jobs, watches, Pub/Sub, or auto-submit behavior.

## Current Repo Truth

- NCKH Phase 1 is completed for OAuth link and Google Forms read/import.
- NCKH Phase 2 is completed for model, variables, and observed question mappings.
- NCKH Phase 3 is completed for backend-only relations, node positions, and deterministic hypothesis output.
- Phase 4 is the next proposed NCKH phase.
- Phase 4 implementation remains approval-gated until contract review, DB review, and Google Forms write-scope review are accepted.
- Phase 3 closeout evidence is the dependency baseline for Phase 4 planning.

## Proposed Phase 4 Scope

In scope for this kickoff plan:

- Google Forms API create/update orchestration behind the existing NCKH auth boundary
- request/response DTOs for model-driven form generation
- service workflow for ownership checks, model readiness checks, and Google authorization checks
- persistence changes only if approved in `NCKH_PHASE_4_CONTRACT_DB_FREEZE.md`
- re-import/update of generated Google Form metadata and question metadata after successful Google API write
- backend tests and authenticated HTTP smoke for the approved route surface

## Out Of Scope

Not allowed in Phase 4:

- Google Sheets API response pull
- response collection
- normalization or dataset generation
- CSV/Excel/SPSS export
- React Flow or broader NCKH frontend expansion
- credit deduction or pricing
- NCKH admin UI
- background jobs, watches, Pub/Sub, or scheduled sync
- automatic response submission
- deleting Google Forms not created or owned through the approved flow
- AI-generated questionnaire text unless separately approved

## Entry Gates

Before any implementation worker starts, confirm all of the following:

1. The user explicitly approves opening NCKH Phase 4 implementation.
2. Contract review is completed for routes, DTOs, action values, response shape, and error behavior.
3. DB review is completed for any new fields, relationships, indexes, migrations, and rollback behavior.
4. Google Forms write scope is explicitly approved for Phase 4 runtime configuration.
5. The flow does not silently pull in Phase 5-8 concerns.
6. `docs/ai/nckh` and `docs/vi/nckh` stay synced for any contract or phase-state change.

## Decisions To Freeze Before Code

1. Google write authorization
   - Candidate scope: `https://www.googleapis.com/auth/forms.body` for creating and updating Google Form body content.
   - Decision needed: whether the existing Google-link flow may be extended to request write scope or whether a separate re-consent step is required.
   - Deferred: Google Sheets scope until Phase 5.

2. Create vs update behavior
   - Proposed: support `create` first.
   - Proposed: support `update` only for forms imported/linked to the current user and allowed by Google authorization.
   - Decision needed: whether update is limited to forms generated by the app or may update a previously imported form owned by the researcher.

3. Model readiness
   - Proposed: model must belong to the current user.
   - Proposed: model must have at least one variable with observed mappings before generation.
   - Decision needed: whether generation is allowed only while model is `Active`, only while `Draft`, or both.

4. Question generation mapping
   - Proposed: generated questions come from `ObservedQuestionMapping` ordered by variable sort order and mapping sort order.
   - Decision needed: exact Google question type mapping for Likert, Nominal, Ordinal, Scale, and text-like fields.
   - Assumption: Phase 4 does not invent statistical wording; it uses existing mapped question text from imported form questions unless a create-from-variable template is explicitly approved.

5. Persistence
   - Decision needed: whether existing `ResearchForm` fields are enough or new fields are required for generated form tracking.
   - Candidate fields if approved: generated source marker, last generated timestamp, last synced timestamp.
   - Do not add database fields until DB review approves exact names and semantics.

6. Re-import behavior
   - Proposed: after successful create/update, call the same form-structure import path to persist current form metadata/questions.
   - Decision needed: whether stale questions are updated in place, marked inactive, or rejected when mappings still reference them.

## Contract Guardrails

- Do not treat proposed Phase 4 endpoints in `NCKH_API_CONTRACT_GUIDE.md` as final until the freeze is accepted.
- Do not enable Google Forms write scope without explicit approval and runtime configuration.
- Do not add Google Sheets scope in Phase 4.
- Do not create data collection, normalization, export, credit, pricing, admin, or frontend behavior in Phase 4.
- Do not delete third-party Google Forms.

## Suggested Delivery Passes

### Pass 0 - Contract And DB Freeze

Goal:

- lock route surface, request/response shape, action values, Google scope, persistence changes, ownership rules, and re-import behavior

Expected outputs:

- accepted `NCKH_PHASE_4_CONTRACT_DB_FREEZE.md`
- accepted `NCKH_PHASE_4_SINGLE_APPROVAL_PACKET.md`

### Pass 1 - Google Forms Generation Service

Goal:

- add only the service/integration behavior needed for approved create/update workflow

Stop conditions:

- Google scope or ownership rules are unclear
- implementation needs Google Sheets, response collection, or frontend behavior

### Pass 2 - API Surface

Goal:

- implement the approved `/api/v1/nckh` route surface for form generation/update

Stop conditions:

- route shape differs from the accepted freeze
- action semantics are unclear

### Pass 3 - Persistence And Re-Import

Goal:

- apply only approved database changes and re-import behavior after successful Google Forms write

Stop conditions:

- re-import would break existing mappings without an approved stale-question policy

### Pass 4 - Validation And Closeout Prep

Minimum validation target:

- `dotnet build FormAutoHub.sln -c Release`
- `dotnet test tests/FormAutoHub.Tests/FormAutoHub.Tests.csproj -c Release`
- `dotnet ef database update` if migrations are added
- authenticated HTTP smoke for the approved route surface
- live Google Forms smoke only when valid credentials/scopes are available
- log inspection after smoke checks
